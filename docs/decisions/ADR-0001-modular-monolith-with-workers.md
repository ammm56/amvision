# ADR-0001 模块化单体与独立 Worker

## 背景

项目需要同时覆盖数据管理、训练、转换、部署、推理、流程编排、插件扩展和外部系统集成。

如果一开始拆成大量微服务，会显著提高本地部署复杂度、调试成本、运行维护门槛和跨模块演进成本，不符合本项目“本地优先、工程落地优先”的阶段目标。

## 决策

采用“模块化单体 + 独立 worker”的总体架构。

- backend-service 作为统一服务入口承担元数据、任务编排和公开接口
- 训练、推理、转换和流程执行通过独立 worker 或执行器承担
- 模块边界优先通过 domain、application、contracts 和稳定接口维持，而不是通过早期强制的网络微服务拆分维持

## 备选方案

### 全微服务架构

- 优点：边界隔离显式、未来独立扩缩容更灵活
- 缺点：本地部署、调试、排障和版本协同成本过高，不适合作为项目早期主架构

### 单进程大一体应用

- 优点：实现简单、部署直接
- 缺点：重任务与接口处理难以隔离，训练/推理/转换会直接拖累 API 服务稳定性

## 影响

- 有利于保持部署简单和开发效率
- 需要通过 QueueBackend、任务状态模型和稳定接口来保证模块边界不会塌陷
- 后续若有充分业务压力，仍可沿现有边界演进为更细粒度服务

## 后续动作

- 持续维护 backend-service、QueueBackend、状态回写和 workers 的明确边界
- 如未来拆分服务，应优先沿既有接口和对象边界演进，而不是重新打散模型
