# ADR-0003 插件优先扩展模型

## 背景

项目需要面对行业差异、客户定制、协议集成、硬件桥接、自定义后处理、外部触发与结果上报等高度变化的需求。

如果把这些能力持续写入核心平台，会快速导致 core 膨胀、边界失控、版本回滚困难和现场定制难管理。

## 决策

采用插件优先扩展模型。

- 场景化能力优先通过插件实现，而不是直接进入核心模块
- 插件体系向 ComfyUI custom nodes 与 workflow 的灵活性看齐，但保留版本、权限、超时、禁用和回滚管理
- 自定义节点、事件 hook、外部触发器、完成回调、数据上报器和后处理器都纳入插件能力模型

## 备选方案

### 主要通过核心代码内建扩展

- 优点：短期实现直接
- 缺点：长期演进和客户定制会持续侵蚀核心模块边界

### 仅支持流程节点扩展，不支持 trigger/hook/回调类扩展

- 优点：系统更简单
- 缺点：无法覆盖大量现场实际需要的外部触发、结果上报和模块衔接需求

## 影响

- 提高了扩展灵活性和客户定制承接能力
- 需要额外建设 manifest、capability、trigger/hook、权限与兼容性管理体系
- 需要在 backend-service 中引入插件生命周期、能力索引和事件点管理

## 后续动作

- 持续维护 plugin-system、manifest-capabilities 和 triggers-hooks 文档
- 为插件提供版本兼容策略、示例模板和审计要求
