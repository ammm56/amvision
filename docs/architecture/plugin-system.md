# 插件系统架构

## 文档目的

本文档用于定义平台插件系统的定位、边界、类型、生命周期和与节点编辑器的关系。

本文档回答的问题是“哪些能力应进入核心平台，哪些能力应通过插件扩展”，以及“如何让插件机制向 ComfyUI 的 custom nodes 和节点编辑体验看齐，同时满足工业场景下的版本、权限、回滚和稳定性要求”。

## 设计目标

- 让核心平台保持稳定、可部署、可回滚，不把场景化需求持续堆入 core
- 让硬件接入、协议集成、模块连接和特殊处理逻辑优先通过插件扩展
- 让流程编排和节点编辑能够把核心节点与插件节点统一纳入同一张执行图
- 让外部触发、完成后数据上报、自定义回调和结果后处理等逻辑能够以插件独立实现与加载
- 让欠缺能力、新行业适配和客户定制能力能够通过插件持续补齐
- 在插件自由度和工业现场可控性之间保持平衡

## 核心原则

- 核心平台负责公开契约、任务编排、版本治理和插件生命周期管理
- 场景化能力优先通过插件扩展，而不是直接扩散到核心模块
- 核心平台默认不内置相机、PLC、传感器、机械臂等硬件直连驱动
- 如确有现场直连需求，应通过受控插件在独立边界中实现，并接受 manifest、权限、超时、禁用和回滚约束
- 插件节点、核心节点和流程模板在节点编辑器中应作为统一的一等公民展示和编排
- 插件能力模型和节点注册机制向 ComfyUI custom nodes 的灵活性看齐，但不能牺牲工业场景的稳定性和可追溯性
- 自定义节点、事件 hook、外部触发器、完成回调、数据上报器和后处理器都应视为插件能力，而不是核心代码中的临时分支

## 插件系统在整体架构中的位置

- backend-service 负责发现插件、注册 manifest、管理启停状态、记录版本和暴露节点目录
- workers 负责在运行时环境中执行插件逻辑，并处理节点输入输出契约
- frontend/web-ui 负责读取节点目录、配置 schema 和分类信息，在节点编辑器与配置面板中渲染插件能力
- shared-contracts 负责插件 manifest、节点定义、能力描述和输入输出 schema 的稳定契约
- runtimes 负责插件运行环境和依赖隔离边界
- packaging 负责基础插件、可选插件包和发布形态的装配策略
- 插件系统同时承担外部系统、内部模块和相关数据对象之间的可配置连接平面

## 插件类型

### 流程节点插件

- 提供模型节点、传统视觉节点、控制节点、条件节点和工具节点
- 参与统一流程编排和节点执行图
- 通过输入输出 schema 和参数 schema 接入执行器与节点编辑器
- 可以像 ComfyUI custom nodes 一样承载客户定制处理逻辑，并与其他节点自由连接

### 后处理插件

- 提供检测结果过滤、规则判定、尺寸测量、缺陷归并和结果转换能力
- 与模型推理结果、传统视觉结果和外部结果处理链路组合使用
- 可在任务完成后执行结果整理、二次计算、结构转换和数据上报前预处理

### 协议适配插件

- 提供与上位机、采集系统、MES、PLC 网关、设备代理系统等外部系统的协议交互能力
- 处理请求接入、状态订阅、结果回传和联动触发
- 可定义特定外部触发方式、回调格式、完成通知和定制数据上报逻辑

### 事件与回调插件

- 提供事件 hook、任务完成回调、状态变化联动和数据上报能力
- 可订阅训练、转换、部署、推理和流程执行过程中的关键事件
- 用于在不修改核心链路的前提下补充客户定制通知、审计、分发和回写逻辑

### 硬件桥接插件

- 作为独立可选扩展实现相机、PLC、传感器、机械臂等硬件直连能力
- 不属于核心平台默认能力，必须在插件边界内独立实现和治理
- 与核心平台只通过受控插件契约交互，不把硬件 SDK 和驱动逻辑渗透进 core

### 模块连接插件

- 提供跨模块连接能力，例如训练结果自动注册、部署完成自动回调、推理结果自动转流程节点输入等
- 用于补足后续业务链路的特殊衔接逻辑和行业特化流程
- 可连接外部接口、内部模块、流程节点和数据对象，形成可配置的衔接链路

### UI 扩展插件

- 为节点编辑器、配置面板和结果查看器提供附加的节点描述、参数面板、表单 schema 或前端展示元数据
- 不直接破坏前端主结构，通过声明式 schema 或受控扩展点接入

## 插件包结构建议

```text
plugins/
└─ <plugin-name>/
   ├─ manifest.yaml
   ├─ backend/
   │  ├─ entry/
   │  ├─ nodes/
   │  ├─ hooks/
   │  └─ adapters/
   ├─ schemas/
   │  ├─ config/
   │  ├─ inputs/
   │  ├─ outputs/
   │  └─ ui/
   ├─ assets/
   └─ docs/
```

## manifest 最低要求

- plugin id
- version
- category 和 capabilities
- entrypoints
- config schema
- input/output schema
- trigger points 和 hook points
- timeout
- permission scope
- dependency constraints
- compatibility range
- enable/disable default policy

## 生命周期

### 1. 发现

- backend-service 在插件目录、本地插件包或后续插件仓库中发现可用插件
- 校验 manifest 完整性、版本兼容性和依赖边界

### 2. 注册

- 将插件元数据、节点定义、schema 和启用状态注册到平台
- 为前端节点编辑器和执行器生成统一节点目录
- 为触发器、hook、回调器和数据上报器建立可检索的能力索引

### 3. 启用

- 允许按插件、按版本、按节点类别启用
- 启用前进行权限校验、依赖检查和运行时兼容性校验

### 4. 执行

- workers 按节点契约、超时和权限边界执行插件逻辑
- 结果和错误统一回到后端服务状态流中
- 插件可在受控事件点执行触发、后处理、结果转换、回调分发和数据上报逻辑

### 5. 升级

- 升级以插件版本为单位进行，不覆盖历史版本记录
- 节点 schema 或行为变化需要显式版本化并提供迁移说明

### 6. 禁用与回滚

- 插件可被按版本禁用或快速回滚
- 回滚后需要恢复节点目录、流程模板兼容性和运行时引用关系

## 节点编辑器对齐 ComfyUI 的方向

- 节点编辑器需要统一展示核心节点与插件节点
- 插件节点应支持分类、搜索、图标、说明、参数 schema 和输入输出端口声明
- 流程模板应像 ComfyUI workflow 一样能保存节点图结构、参数状态和版本引用
- 自定义节点的注册、卸载和升级不应要求修改核心前端代码结构
- 与 ComfyUI 对齐的是“节点和插件的灵活扩展模型”，不是照搬其无约束运行方式
- 除显式流程节点外，事件 hook、触发器和回调器也应能作为受控扩展能力被管理和编排

## 与核心模块的关系

### backend-service

- 管理插件注册、版本、启用状态和节点目录发布
- 记录插件与流程模板、部署实例、任务类型之间的引用关系
- 负责事件点、触发点和回调点的注册、鉴权与治理

### workers

- 提供插件执行容器、节点调度、错误收敛和超时控制
- 确保插件输入输出严格遵循 schema 契约
- 负责在任务完成、状态变化和结果产出等时机调用对应插件逻辑

### frontend/web-ui

- 读取插件节点目录并渲染节点面板、参数表单和流程图
- 为插件配置、状态和错误提供受控展示界面

### shared-contracts

- 定义 manifest、节点定义、参数 schema、输入输出 schema 和 capability descriptor
- 避免每类插件自行发明一套不兼容协议
- 定义事件 hook、触发器、回调载荷和数据上报格式的公共契约

## 安全与治理要求

- 插件必须声明 capability scope，避免无限制调用平台能力
- 硬件桥接插件和协议适配插件必须声明额外权限和外部依赖
- 插件必须支持 timeout、disable、health check 和版本回滚
- 插件错误不能直接拖垮后端服务主链路，应通过任务和状态流隔离处理
- 插件日志、错误和版本必须可审计
- 具有外部触发、数据上报和回调能力的插件必须声明可访问的端点、数据范围和事件范围

## 哪些能力优先放进插件

- 行业特定协议
- 客户定制回调和联动逻辑
- 外部触发入口和完成后的数据上报逻辑
- 硬件直连与厂商 SDK 封装
- 特定视觉后处理逻辑
- 模块之间的特殊衔接规则
- 自定义节点、节点组和参数面板

## 哪些能力应留在核心平台

- 任务模型和状态流
- 数据集、模型、部署、流程模板的核心对象模型
- 插件生命周期管理和版本治理
- 节点执行基础框架和流程模板基础格式
- 统一 API、WebSocket 和审计能力
